    Eventos de jugador

    En este tema, aprenderá acerca de los diversos tipos de eventos asociados con Brightcove Player.

    Resumen

    El Brightcove Player tiene muchos eventos para que usted pueda controlar la reproducción de vídeo. En este tema se proporciona información general sobre los diferentes tipos de eventos.

    Puede encontrar una lista completa de los eventos de Brightcove Player en el documento de referencia de la API Métodos/Eventos del Reproductor . Este documento refleja los eventos que forman parte del código real que compone el reproductor.

    Eventos en los medios

    El documento HTML Living Standard es un gran recurso para obtener información sobre el desarrollo de HTML y las API necesarias para las aplicaciones web. Este es un documento «vivo» que se mantiene actualizado por el Web Hypertext Application Technology Working Group (WHATWG), que es una comunidad creciente de personas interesadas en evolucionar la web.

    El Brightcove Player se ejecuta en la parte superior del marco HTML, lo que desencadena los siguientes eventos multimedia.

    Aquí hay algunos términos asociados con eventos en este documento:

    bloqueado

    Un MediaController se considera bloqueado si la reproducción se ha pausado. Un elemento multimedia se bloquea si su controlador es un controlador de medios bloqueado.

    MediaController

    Un MediaController es un objeto que coordina la reproducción de varios elementos multimedia.

    elemento multimedia

    Un elemento multimedia presenta datos de audio, o datos de vídeo y audio, al usuario.

    esclavos

    Por defecto, cada elemento multimedia tiene no MediaController. Cuando los elementos multimedia están asociados con un MediaController, se dice que son esclavos del controlador. El controlador modifica la velocidad de reproducción y el volumen de cada uno de los elementos multimedia esclavos a él. Si uno de los elementos esclavos se detiene inesperadamente, el controlador detendrá los otros elementos esclavos.

    Almacenamiento en búfer y eventos de QoS

    Aquí hay un subconjunto de eventos relacionados con el almacenamiento en búfer y la calidad del servicio (QoS):

    Nombre del evento Descripción
    progress Almacenamiento en búfer (carga) de datos de vídeo
    waiting Esperando momentáneamente los datos de vídeo solicitados
    stalled Almacenamiento en búfer estancado
    error Error al cargar el vídeo
    playing Reproducción reanudada tras pausa o retraso en la descarga
    ratechange La velocidad de reproducción ha cambiado (puede ser manual o automática)
    En IE11/Edge Windows 10, el waiting evento nunca se distribuye cuando hay almacenamiento en búfer.

    Eventos de inicio

    Hay una serie de eventos que ocurren cuando el reproductor se inicializa y se prepara para reproducir un vídeo. Se enumeran aquí en el orden en que se enviarán:

    • loadstart: Se distribuye cuando se inicializa el reproductor, y si se reinicializa en el caso de darle una nueva fuente para jugar

      Dado que el loadstart evento puede ocurrir antes de que el reproductor esté listo, es recomendable establecer el detector de eventos para este evento antes de llamar a la ready() función del jugador. Vea el CodePen a continuación.
    • loadedmetadata: Se distribuye cuando el reproductor tiene información inicial de duración y dimensión, es decir, cuando se descarga el primer segmento. Para vídeos en directo, el loadedmetadata evento no se enviará hasta que el usuario haga clic en Reproducir. Esto se debe a que los vídeos en directo no comienzan a descargar segmentos hasta que el usuario hace clic en Reproducir.
    • loadeddata: Se distribuye cuando el reproductor ha descargado datos en la posición de reproducción actual

    El siguiente CodePen muestra los eventos de inicio que se están escuchando y distribuyendo. Tenga en cuenta que si pasa el ratón sobre el CodePen, puede hacer clic en el botón Volver a ejecutar ubicado en la parte inferior derecha para ver el envío de los eventos de nuevo. Si desea experimentar con el código, haga clic en el enlace Editar en CODEPEN en el encabezado para desplazarse al entorno de edición. Puede hacer clic en el botón JS para ver el JavaScript que agrega los detectores de eventos.

    Consulte la demostración de eventos de inicio de pluma de Brightcove Learning Services (@bcls ) en CodePen.

    Nota: Es posible que en la primera carga de esta página no se vea el loadstart evento. Esto se debe a que ese evento podría distribuirse antes de que el CodePen esté completamente representado en la página. Haga clic en el botón Volver a ejecutar y verá los tres eventos distribuido.

    ready (), método

    El ready() método tiene una doble personalidad en el sentido de que parece un evento, pero lo usas como un método. Este método/evento significa que el jugador está listo para recibir comandos.

    El ready() método difiere de los detectores de eventos en que si el ready evento ya ha ocurrido, desencadenará el ready() método inmediatamente. A menudo verá el método utilizado de la siguiente manera para proporcionar el punto de partida para el desarrollo con Brightcove Player:

    videojs.getPlayer('myPlayerID').ready(function(){
  var myPlayer = this;
});
    Nota: Es importante darse cuenta de que si el reproductor tiene que cambiar las tecnologías de reproducción, digamos de HTML a Flash, para reproducir un vídeo, los controladores de eventos de ready método/evento se volverán a ejecutar una vez que la nueva tecnología de reproducción esté funcionando.

    ready () vs. on ('loadedmetadata',...)

    Tenga en cuenta que el uso de ready() funciones correctamente si desea interactuar con el reproductor, por ejemplo, agregue un plugin mediante programación. Si desea interactuar inmediatamente con el vídeo, por ejemplo play(), usar, el fragmento de código anterior no siempre funcionará correctamente. Un mejor enfoque sería escuchar para loadedmetadata interactuar con el video, por ejemplo:

    videojs.getPlayer('myPlayerID').on('loadedmetadata',function(){
  var myPlayer = this;
  myPlayer.play();
});

    En resumen, para interactuar con el reproductor puede usar esto:

    videojs.getPlayer('myPlayerID').ready()

    Si desea interactuar inmediatamente con el vídeo en el reproductor, utilice esto:

    videojs.getPlayer('myPlayerID').on('loadedmetadata',function(){})
    Nota: La razón por la que debe esperar a que se distribuya el loadedmetadata evento es que garantiza que el catalog ha tenido tiempo para recuperar el vídeo antes de intentar reproducirse.

    Eventos a pantalla completa

    El jugador emite un fullscreenchange evento cuando se cambia hacia o desde el modo de pantalla completa. El mismo evento se transmite si el jugador va a pantalla completa o regresa al modo normal, Si es importante para usted saber lo que sucedió, utilice el player.isFullscreen() método para determinar si el jugador está actualmente en modo de pantalla completa.

    El Codepen que aparece a continuación ilustra cómo hacerlo.

    Consulte los eventos de pantalla completa Pen Brightcove Player de Brightcove Learning Services (@bcls ) en CodePen.

    Eventos publicitarios

    Para incluir bibliotecas publicitarias y funciones para su reproductor, puede utilizar el plugin IMA4 o el plugin FreeWheel. Esto se construye sobre el marco de anuncios de videojs (videojs-contrib-ads). Ambos complementos publicitarios pueden enviar los eventos publicitarios que se muestran en la tabla siguiente. Cada uno también tiene eventos específicos para la implementación.

    Evento Se envía cuando:
    solicitud de anuncios A petición de datos publicitarios.
    carga de anuncios Cuando los datos del anuncio estén disponibles tras una solicitud de anuncio.
    ads-ad-iniciado Ha empezado a reproducirse un anuncio.
    ads-ad-finalizado Un anuncio ha terminado de reproducirse.
    pausa de anuncios Un anuncio está en pausa.
    anuncios de reproducción Un anuncio se reanuda a partir de una pausa.
    ads-primer cuartil El anuncio ha reproducido el 25% de su duración total.
    punto medio de anuncios El anuncio ha reproducido el 50% de su duración total.
    ads-tercer cuartil El anuncio ha reproducido el 75% de su duración total.
    anuncios y clic Un espectador hizo clic en el anuncio de reproducción.
    ads-volumechange Se ha cambiado el volumen del anuncio de reproducción.
    ads-pod-iniciada Ha comenzado el primer anuncio de un pod de anuncios lineal (un grupo secuenciado de anuncios).
    ads-pod-finalizado El último anuncio de un pod de anuncios lineal (un grupo secuenciado de anuncios) ha finalizado.
    ads-allpods-completado Todos los anuncios lineales han terminado de reproducirse.

    evento bc-catalog-error

    Es posible que el manejo de errores en la ready() sección normal en el bloque de script pueda causar problemas. Por ejemplo, puede suceder que el bc-catalog-error evento se pueda enviar antes de que el jugador esté listo, y si escucha el error en la ready() sección, no podrá manejar el error. Este problema puede producirse cuando se utiliza el filtrado geográfico, un vídeo no se publica, un vídeo está fuera del rango de programación o en una cuenta diferente. Para obtener una discusión completa y ejemplos de manejo del bc-catalog-error evento, consulte el documento Catálogo del Reproductor .

    progreso y timeupdate

    Puede haber confusión acerca de las diferencias entre progress y timeupdate eventos. El Brightcove Player se construye sobre vídeo HTML5, y en ese estándar el progress evento se distribuye cuando el navegador está descargando datos. El timeupdate evento se distribuye cuando la posición de reproducción actual ha cambiado.

    Esto puede resultar confuso para los usuarios del Smart Player de Brightcove, ya que en esa API el progress evento hace lo que timeupdate hace en la API de Brightcove Player.

    Precaución al eliminar los oyentes de timeupdate

    En algunos casos, es posible que desee eliminar un detector de eventos que ha agregado al timeupdate evento. En el ejemplo del Brightcove Player se muestra un caso de uso: Registrarse en Reproducir después de la vista previa del documento. En esta situación, desea que el reproductor se detenga en algún intervalo de tiempo, y que está utilizando timeupdate para la comprobación de tiempo y, a continuación, utilice una definición de función de controlador de eventos anónima. Solo desea hacer una pausa una vez, por lo que debe eliminar el detector de eventos. No quieres simplemente hacer:

    myPlayer.off('timeupdate');

    Por supuesto, esto elimina TODOS los oyentes de eventos a timeupdate, y entre otros problemas al hacer esto evitará que el depurador de tiempo avance. Lo que se necesita hacer es crear una función usando la sintaxis de definición de función normal (una declaración de función), luego eliminar SOLO el oyente de eventos. El código abreviado se muestra aquí:

     // Add the event handler
myPlayer.on('timeupdate', timeupdateHandler);

// Handle the event then remove JUST this event listener on timeupdate
function timeupdateHandler(evt) {
  ...
  myPlayer.off('timeupdate', timeupdateHandler);
}

    Eventos de análisis

    Se activa un evento cada vez que se envía un baliza al recopilador de datos de Analytics. Puede escuchar cualquier baliza o específica.

    Escuchar todas las balizas

    Puede realizar un seguimiento de todas las balizas de análisis enviadas escuchando el analytics_request evento:

    player.on('analytics_request', function(e) {
  videojs.log('sent a(n) ' + e.params.event + ' beacon!', e.params);
});

    Escuchar balizas específicas

    Puede realizar un seguimiento de las balizas de análisis específicas enviadas escuchando el analytics_request_{beacon_name} evento.

    {beacon_name} es el nombre de cualquiera de los eventos del reproductor que se envían al recopilador de datos. Para obtener una lista completa de estos eventos, consulte Referencia de Data Collection API.

    Ejemplo

    player.on('analytics_request_video_impression', function(e) {
  videojs.log('sent an impression beacon!', e.params);
});

    No hay soporte para el encadenamiento de métodos

    A partir de la versión 6 de Brightcove Player, ya no se admite el encadenamiento con eventos. Esto significa que NO PUEDES hacer esto:

    player.on(event, function () {})
.on(event, function () {})
.on(event, function () {})
.on(event, function () {});
