    Complemento HLS

    El complemento HLS es un plugin de Brightcove Player que se basa en las extensiones de fuente multimedia (MSE) de W3C para reproducir vídeo HTTP Live Streaming (HLS) en plataformas que no lo admiten de forma nativa. El plugin HLS se incluye automáticamente en el reproductor.

    Conceptos básicos del complemento HLS

    Los siguientes puntos le ayudan a entender y usar el complemento HLS:

    • Como se menciona en la frase inicial de este documento, el plugin se basa en las extensiones de fuente de medios (MSE) de W3C. MSE es una especificación W3C que permite a JavaScript enviar flujos de bytes a códecs multimedia dentro de navegadores web que admiten vídeo HTML5. Entre otros usos posibles, esto permite la implementación de código de búsqueda previa y almacenamiento en búfer del lado del cliente para streaming de medios completamente en JavaScript.
    • Con el plugin puede usar contenido de vídeo HLS (m3u8) en el reproductor. Por ejemplo, podría crear un reproductor usando esta configuración para la sección multimedia: 
      "media":{
    "sources": [{
        "src": "http://example.com/video.m3u8",
        "type": "application/x-mpegURL"
    }]
}
    • El intercambio de recursos de origen cruzado (CORS) puede ser un problema al usar HLS. Para obtener más información sobre el uso de CORS, consulte la guía CORS.
    • HLS no es compatible con versiones de IE anteriores a IE11.

    Resumen

    HTTP Live Streaming (HLS) se ha convertido en un estándar de facto para la transmisión de vídeo en dispositivos móviles gracias a su soporte nativo en iOS y Android. Aunque, hay una serie de razones independientes de la plataforma para recomendar el formato:

    • Admite la selección de velocidad de bits adaptable (impulsada por el cliente)
    • Entregado a través de puertos HTTP estándar
    • Formato de manifiesto simple basado en texto
    • No se requieren servidores de transmisión patentados

    Desafortunadamente, todos los principales navegadores de escritorio excepto Safari faltan soporte HLS. Esto deja a los desarrolladores web en la desafortunada posición de tener que mantener copias alternativas del mismo vídeo y potencialmente tener que renunciar por completo al vídeo basado en HTML para proporcionar la mejor experiencia de visualización de escritorio.

    Este plugin soluciona la situación proporcionando un polyfill para HLS en navegadores que tienen extensiones de origen multimedia o soporte Flash. Puede implementar una única secuencia HLS, codificar con las API de vídeo HTML5 normales y crear una experiencia de vídeo rápida y de alta calidad en todas las grandes categorías de dispositivos web.

    Actualmente, en este momento no hay soporte para:

    • Pistas alternativas de audio y vídeo
    • Códecs de segmentos distintos de H.264 con audio AAC
    • Internet Explorer < 11
    Nota: Para obtener la máxima compatibilidad con todas las implementaciones de MSE, Brightcove recomienda configurar audio_bitrate a no menos de 48 en los perfiles de ingesta.

    Opciones

    Hay varias opciones que puede usar para configurar el plugin HLS.

    Con Credenciales

    Tipo:boolean

    Se puede utilizar como:

    • una opción de origen
    • una opción de inicialización

    Cuando el withCredentials la propiedad está establecida en true , todas las solicitudes XHR de manifiestos y segmentos tendrían withCredentials ajustado a true también. Esto permite almacenar y pasar cookies desde el servidor en el que viven los manifiestos y segmentos. Esto tiene algunas implicaciones en CORS porque cuando se establece, el Access-Control-Allow-Origin encabezado no se puede establecer en * , además, los encabezados de respuesta requieren la adición de Access-Control-Allow-Credentials encabezado que se establece en true. Consulte el artículo de HTML5Rocks para obtener más información.

    Problema conocido: NO withCredentials funciona con Safari en el escritorio o dispositivos.

    Puede configurar el plugin utilizando la API de administración del reproductor utilizando un método HTTP, como se muestra a continuación:PATCH

    curl \
    --header "Content-Type: application/json" \
    --user YOUR_EMAIL \
    --request PATCH \
    --data '{ "hls": { "withCredentials": true } }' \
https://players.api.brightcove.com/v2/accounts/YOUR_ACCOUNT_ID/players/YOUR_PLAYER_ID/configuration

    También puede establecer la withCredentials opción por fuente, en lugar de por jugador como se acaba de mostrar. Por ejemplo, al configurar el origen puede incluir withCredentials, como se muestra aquí:

    curl \
  --header "Content-Type: application/json" \
  --user $EMAIL \
  --request POST \
  --data '{
      "name": "MySamplePlayer",
      "configuration": {
        "media": {
          "sources": [{
            "src":"http://solutions.brightcove.com/bcls/assets/videos/Tiger.mp4",
            "type":"video/mp4",
            "withCredentials": true
          }]
        }
      }
    }' \
https://players.api.brightcove.com/v2/accounts/$ACCOUNT_ID/players

    Configuración de tiempo de ejecución

    Puede configurar withCredentials en tiempo de ejecución. Verá a continuación dos implementaciones:

    • Uso de player.hls.xhr.beforeRequest
    • Uso de player.src()

    En el siguiente código está utilizando player.hls.xhr.beforeRequest para asignar una función que se llamará con un objeto que contiene opciones que se utilizarán para crear la xhr solicitud. En este ejemplo sólo withCredentials se está configurando.

    if (videojs.Hls) {
  videojs.Hls.xhr.beforeRequest = function (options) {
    options.withCredentials = true;
  }
}

    También puede establecer las withCredentials opciones al configurar la fuente de vídeo. Usa el player.src() método, como se muestra aquí:

    Existe un problema conocido actual de que esta opción de configuración no funciona cuando se utiliza Firefox con anuncios IMA4. Puede utilizar player.hls.xhr.beforeRequest para evitar este problema.
    player.src({
  src: 'https://adomain.com/bipbop_16x9_variant.m3u8',
  type: 'application/x-mpegURL',
  withCredentials: true
});

    EnableLowInitialPlaylist

    Tipo:boolean

    Predeterminado: undefined, excepto cuando se ve un navegador en un dispositivo Android; a continuación, se establece en true. Puede cambiar este comportamiento para dispositivos Android, parcheando el reproductor, como se muestra a continuación, con un valor de false.

    Se puede utilizar como:

    • una opción de inicialización

    Cuando enableLowInitialPlaylist se establece en true, se usará para seleccionar la lista de reproducción de velocidad de bits más baja inicialmente. Esto ayuda a reducir el tiempo de inicio de la reproducción.

    Puede configurar el plugin utilizando la API de administración del reproductor utilizando un método HTTP, como se muestra a continuación:PATCH

    curl \
    --header "Content-Type: application/json" \
    --user YOUR_EMAIL \
    --request PATCH \
    --data '{ "hls": { "enableLowInitialPlaylist": true } }' \
https://players.api.brightcove.com/v2/accounts/YOUR_ACCOUNT_ID/players/YOUR_PLAYER_ID/configuration

    Propiedades de tiempo de ejecución

    En general, puede acceder al objeto HLS de esta manera:

    • Brightcove Player v5: player.hls
    • Brightcove Player v6: player.tech().hls
      Tenga en cuenta que la llamada player.tech() registrará una advertencia en la consola, ya que solo se recomienda para usuarios avanzados. Puede pasar player.tech(true) para suprimir la advertencia.

    player.hls.playlists.master

    Tipo:object

    Objeto que representa la lista de reproducción principal analizada. Si una lista de reproducción multimedia se carga directamente, se creará una lista de reproducción maestra con una sola entrada.

    player.hls.playlists.media

    Tipo:function

    Función que se puede utilizar para recuperar o modificar la lista de reproducción multimedia activa actualmente. La lista de reproducción multimedia activa se hace referencia cuando es necesario descargar datos de vídeo adicionales. Llamar a esta función sin argumentos devuelve el objeto de lista de reproducción analizado para la lista de reproducción multimedia activa. Llamar a esta función con un objeto de lista de reproducción desde la lista de reproducción maestra o una cadena URI tal como se especifica en la lista de reproducción maestra iniciará una carga asincrónica de la lista de reproducción multimedia especificada. Una vez recuperado, se convertirá en la lista de reproducción multimedia activa.

    player.hls.bandwidth

    Tipo:number

    El número de bits descargados por segundo en la última descarga del segmento. Este valor es utilizado por la implementación predeterminada de selectPlaylist para seleccionar una velocidad de bits adecuada para jugar. Antes de descargar el primer segmento de vídeo, es difícil estimar el ancho de banda con precisión. La tecnología HLS utiliza una heurística basada en los tiempos de descarga de la lista de reproducción para hacer esta estimación por defecto. Si tiene una fuente más precisa de información de ancho de banda, puede anular este valor tan pronto como la tecnología HLS se haya cargado para proporcionar una estimación inicial del ancho de banda.

    Player.hls.stats.bytesReceived

    Tipo:number

    El número total de bytes de contenido descargados por la tecnología HLS.

    Player.hls.SelectList

    Tipo:function

    Función que devuelve el objeto de lista de reproducción multimedia que se va a utilizar para descargar el siguiente segmento. Es invocado por el plugin inmediatamente antes de que se descargue un nuevo segmento. Puede anular esta función para proporcionar su lógica de transmisión adaptable. Sin embargo, debe asegurarse de devolver un objeto de lista de reproducción multimedia válido que esté presente en player.hls.playlists.master.

    Eventos

    cargadometadatos

    Se activa después de descargar la primera lista de reproducción multimedia para una transmisión.

    lista de reproducción cargada

    Se disparó inmediatamente después de que se haya descargado una nueva lista de reproducción principal o multimedia. De forma predeterminada, el plugin solo descarga listas de reproducción según sean necesarias.

    mediachange

    Se activa cuando una nueva lista de reproducción se convierte en la lista de reproducción multimedia activa. Tenga en cuenta que el cambio de calidad de representación real no se produce simultáneamente con este evento; se debe solicitar un nuevo segmento y primero se debe agotar el búfer existente.

    Recargar fuente en caso de error

    Cuando se utiliza el plugin HLS hay un método al que puede llamar que recargará la fuente en su momento actual cuando el reproductor emita un error. Para activar esta función, debe llamar al reloadSourceOnError() método. El siguiente vídeo corto muestra el método en acción. Todo el código que se muestra en el video se describe más adelante en esta sección.

    Nota: Aunque originalmente destinado a la transmisión en vivo, también reloadSourceOnError() funcionará con video bajo demanda, como se muestra en el video.

    La sintaxis del reloadSourceOnError() método es la siguiente:

    reloadSourceOnError(optionsObject)

    El opcional optionsObject tiene las siguientes propiedades:

    Propiedad Tipo de datos Valor predeterminado Descripción
    errorInterval Número 30 La cantidad mínima de tiempo (en segundos) que tiene que pasar entre dos errores para que la recarga se active. Por ejemplo, si establece el tiempo en 10, cada vez que se produce un error, la función comprobará si se ha producido una recarga hace menos de 10 segundos. Si ha pasado menos del intervalo de tiempo, NO volverá a cargar el origen. (Esto es para garantizar que el contenido con un error no se recargue constantemente). Si ha pasado más tiempo del intervalo especificado, el vídeo se vuelve a cargar en el punto en que se produjo el error.
    getSource() Función Recupera el origen actual Función que se llama para obtener un objeto de origen para cargar o recargar. De forma predeterminada, obtiene la fuente actual del reproductor.

    In-Manifest WebVTT

Los videos deben ser ingeridos con WebVTT in-manifest considerado. Por ejemplo, Brightcove Dynamic Ingest API puede ingerir vídeos y configurar subtítulos como in-manifest. Consulte el documento Visión general: Dynamic Ingest API para Dynamic Delivery para obtener más detalles.

    Conmutación adaptable

    Selección de copia HLS

    Durante la reproducción, el reproductor cambiará a una representación más alta o inferior basada en un algoritmo. Las entradas a este algoritmo son:

    • ancho de banda disponible
    • Dimensiones del jugador

    Para obtener una explicación completa de la selección de copias, consulte el documento Determinación de la copia que se reproducirá .

    Selección de copia MP4

    Si por alguna razón Brightcove Player no puede reproducir fuentes HLS, volverá a reproducir MP4. En este caso, si ve un video en un dispositivo móvil y reproduce un MP4, el reproductor elegirá el MP4 que tenga una velocidad de bits más cercana a .5 MB/s. Si está en un dispositivo de escritorio o portátil, elegirá la representación MP4 más cercana a 3 MB/s.

    Metadatos en banda

    Brightcove Player reconocerá ciertos tipos de información de etiquetas ID3 incrustadas en una transmisión de vídeo HLS. El estándar ID3 se usó originalmente para proporcionar metadatos sobre pistas de audio MP3. (El acrónimo es de IDentify MP3.) Cuando se encuentra una secuencia con metadatos incrustados, se creará automáticamente una pista de texto de metadatos en banda y se rellenará con señales a medida que se encuentren en la secuencia. Un caso de uso común es que los datos ID3 dirigirían cuando los anuncios deben mostrarse en una transmisión en vivo.

    El estándar ID3 define muchos tipos de fotogramas, pero solo los siguientes dos marcos codificados UTF-8 se asignarán a puntos de referencia y sus valores se establecerán como texto de referencia:

    • WXXX - Marco de enlace de URL definido por el usuario
    • TXXX - Marco de información de texto definido por el usuario

    Las señales se crean para todos los demás tipos de fotogramas y los datos se adjuntan a la señal generada:

    cue.frame.data

    Para obtener más información acerca de las pistas de texto en general, consulte Introducción al elemento Track. Para obtener más información sobre el Brightcove Player y los puntos de referencia, consulte Visualización de anuncios mediante puntosde anuncio.

    608 leyendas

    El complemento HLS de Brightcove admite 608 subtítulos. 608 subtítulos, también conocidos como subtítulos CEA-608, EIA-608 y subtítulos de la línea 21, son un estándar para subtítulos opcionales para las emisiones analógicas de televisión NTSC en los Estados Unidos y Canadá. Los 608 subtítulos se pueden insertar en una transmisión en vivo donde se mezclan en los archivos ts (transmisión de transporte) de HLS.

    Problemas de

    A diferencia de una implementación nativa de HLS, el complemento HLS tiene que cumplir con las políticas de seguridad del navegador. Esto significa que todos los archivos que componen la secuencia deben ser servidos desde el mismo dominio que la página que aloja el reproductor de vídeo o desde un servidor que tenga configurados los encabezados CORS adecuados. Hay instrucciones sencillas disponibles para los servidores web populares y la mayoría de los CDN no deberían tener problemas para activar CORS en su cuenta.

    Errores

    Los errores durante la reproducción de HLS se notificarán utilizando el tipo APPEND_BUFFER_ERR. El mensaje será lo que se recupera del error nativo del navegador. Por ejemplo, se ha superado la cuota.
